## 2. 七大优化维度

### 维度 A：架构瘦身与模块解耦

**问题**: 核心模块膨胀，部分文件触 SLOC 天花板，adapters/__init__.py 承载了过多 verify 逻辑。

#### A1. adapters/__init__.py 拆分 (verify 逻辑外移)

**现状**: `adapters/__init__.py` 845 行，其中 ~350 行是 `_verify_*_live()` / `_check_*_setup()` 函数，混杂了注册声明与验证逻辑。

**方案**:
- 将 `_verify_live` / `_verify_dry_run` / `_check_*_setup` / 所有 helper 函数移至 `publishing/_verify_adapters.py`（或按平台拆为 `publishing/verify/` 子包）
- `__init__.py` 只保留 `register()` 调用表 + `publish()` / `verify_adapter_setup()` 的 thin dispatch
- 目标: `__init__.py` 降到 ~300 行

**验收**: `adapters/__init__.py` < 350 行; 所有 `test_adapter_*` 测试保持绿色; 无公开 API 变更

#### A2. plan_backlinks 进一步分解

**现状**: `cli/plan_backlinks.py` 已是 package（`_engine.py` / `_payload.py` / `_links.py` / `_templates.py` / `_zh_short.py` / `_work_themed.py` / `_banners.py`），SLOC 天花板 1270。

**方案**:
- 审计 `_engine.py` 的实际 SLOC，识别可提取的纯函数
- 将 LLM anchor 生成逻辑从 `_links.py` 提取到 `anchor/llm_bridge.py`（如果尚未分离）
- 将 seed input 解析（JSONL / CSV / sitemap）提取到 `_input.py`

**验收**: 最大单文件 SLOC < 800; `monolith_budget.toml` 天花板不升反降

#### A3. webui_app 路由模块归一化

**现状**: 20 个路由模块，部分可能只有 1-2 个 endpoint。

**方案**:
- 审计各路由模块行数，合并 <50 行的微模块（如 `token_paste.py` + `bind_channel_save.py` → `binding.py`）
- 确保合并后 blueprint 名不变（保持 URL 兼容）

**验收**: 路由模块数降到 12-15; 无 URL 变更; `test_webui_route_contract.py` 保持绿色

#### A4. 提取公共 HTTP 客户端

**现状**: 多个 adapter 独立 `import requests` / 独立构造 headers / 独立 timeout 管理。`http.py` 存在但未被所有 adapter 统一使用。

**方案**:
- 创建 `publishing/_http_client.py`：封装 retry + timeout + SSRF check + UA rotation 的统一 HTTP session
- 逐步迁移 adapter 使用统一客户端（每个 adapter PR 独立）
- 保留 escape hatch: adapter 可直接用 requests 但需要 `# noqa: no-unified-http` 注释

**验收**: ≥60% adapter 使用统一 HTTP 客户端; 无行为变更

### 维度 C：性能与并发

**问题**: 管线完全串行，无连接池，无缓存层。当真实流量从 0 → 两位数时，性能瓶颈将显现。

#### C1. URL 验证并行化

**现状**: `validate-backlinks` 对每个 URL 串行 HTTP check。10 条 × 6-8 links = 60-80 次 HTTP 请求，串行 ~30-60 秒。

**方案**:
- 使用 `concurrent.futures.ThreadPoolExecutor` 并行化 URL 验证
- 默认 workers = min(8, url_count)
- 添加 `--parallel` / `--no-parallel` CLI flag
- 保留 `--no-check-urls` escape hatch

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def verify_urls_parallel(urls: list[str], *, workers: int = 8) -> dict[str, UrlResult]:
    results = {}
    with ThreadPoolExecutor(max_workers=workers) as pool:
        futures = {pool.submit(check_url, url): url for url in urls}
        for future in as_completed(futures):
            url = futures[future]
            results[url] = future.result()
    return results
```

**验收**: 80 URL 验证从 ~40s 降到 ~8s (5x); 所有 `test_validate_*` 测试保持绿色

#### C2. Adapter 连接池

**现状**: 每个 adapter 独立创建 `requests.Session`（或直接用 `requests.post/get`），无连接复用。

**方案**:
- 在 `publishing/_http_client.py`（A4 产出）中使用 `requests.Session` + `HTTPAdapter(pool_connections=10, pool_maxsize=10)`
- Adapter 通过 session 发请求，自动复用 TCP 连接
- 对同一平台的批量发布，连接复用减少 TLS 握手开销

**验收**: 连续发布 10 条到同一平台，TCP 连接数从 ~10 降到 ~1-2

#### C3. 内容抓取缓存

**现状**: `content/fetch.py` 每次调用都发 HTTP 请求。同一 URL 在 plan + validate 阶段可能被 fetch 多次。

**方案**:
- 在 `~/.cache/backlink-publisher/content-cache/` 下按 URL SHA 缓存响应
- TTL: 1 小时（可配置 `[cache] content_ttl_seconds`）
- 添加 `--no-cache` flag 强制刷新
- 使用 `Cache-Control` / `ETag` 尊重上游缓存策略

**验收**: 同一 URL 第二次 fetch < 10ms（命中缓存）; `test_content_fetch.py` 保持绿色

#### C4. WebUI 静态资源优化

**现状**: 零构建原生 ES modules，无 minification / compression。

**方案**:
- 添加可选的 `make optimize-static` 目标：
  - CSS: 使用 `lightningcss` minify
  - JS: 使用 `esbuild` minify（不 bundle，保持 ESM）
  - 图片: 使用 `oxipng` 优化
- 生产模式 (`FLASK_ENV=production`) 自动使用 minified 版本
- 开发模式保持原始文件（方便调试）

**验收**: 静态资源总大小减少 ≥40%; WebUI 首屏加载 < 2s (loopback)

### 维度 D：安全加固

**问题**: 安全基础好（CSRF + loopback + SSRF），但缺少纵深防御。

#### D1. WebUI Rate Limiting

**现状**: 无 rate limit。loopback 绑定是主要防线，但 `ALLOW_NETWORK=1` 时暴露风险。

**方案**:
- 使用 `flask-limiter` 或自实现的 token bucket
- 限制: POST/PUT/PATCH/DELETE ≤ 30 req/min per IP
- 特别限制: `/copilot/ask` ≤ 5 req/min（LLM 调用成本高）
- 超限返回 429 + `Retry-After` header

**验收**: 连续 31 次 POST → 第 31 次返回 429; 正常操作不受影响

#### D2. JSONL 输入大小限制

**现状**: `plan-backlinks` / `validate-backlinks` / `publish-backlinks` 从 stdin 读取无大小限制。

**方案**:
- 添加 `--max-rows` flag（默认 1000）
- 超过限制时 exit 2 + stderr 提示
- 防御: 防止意外喂入 GB 级文件导致 OOM

**验收**: `cat huge.jsonl | plan-backlinks --max-rows 10` 只处理前 10 行 + WARN

#### D3. Credential 文件权限审计工具

**现状**: AGENTS.md 提到部分 state 文件可能仍是 0644。

**方案**:
- 创建 `scripts/audit_credential_permissions.py`
- 扫描 `config_dir` 下所有 `*-state.json` / `*-token.json` / `*-cookies.json` / `credentials.json`
- 报告非 0600 的文件
- 添加 `--fix` flag 自动 `chmod 0600`
- 集成到 `preflight-targets` 或 WebUI `/settings` 的诊断卡片

**验收**: `python scripts/audit_credential_permissions.py` 列出所有非 0600 文件; `--fix` 修复后重跑零报告

#### D4. SSRF 增强: Cloud Metadata 阻断

**现状**: SSRF guard 存在但可能未覆盖所有 cloud metadata endpoint。

**方案**:
- 确保 `_util/net_safety.py` 阻断以下 IP:
  - `169.254.169.254` (AWS/GCP/Azure IPv4)
  - `fd00:ec2::254` (AWS IPv6)
  - `168.63.129.16` (Azure wireserver)
  - `100.100.100.200` (Alibaba Cloud)
- 添加测试覆盖（参数化所有已知 metadata IP）

**验收**: `pytest -m real_ssrf_check` 对所有 metadata IP 返回 blocked

import structlog

logger = structlog.get_logger()
logger.info("publish.succeeded", platform="blogger", article_id="abc123", duration_ms=1234)
# → {"event": "publish.succeeded", "platform": "blogger", "article_id": "abc123", "duration_ms": 1234, "timestamp": "..."}

**验收**: `BACKLINK_LOG_FORMAT=json publish-backlinks ...` 输出合法 JSON 日志; 默认模式保持人类可读

**现状**: 无指标导出。运营者只能从 events.db 手动聚合。

#### E3. 运营状态 Dashboard 增强

**现状**: `/health` 显示 canary / publish-path / scorecard 卡片。

**方案**:
- 添加「管线健康」总览卡片:
